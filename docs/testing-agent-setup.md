# テストエージェント導入計画（案A: MockBukkit + 自動テスト）

## 目的

開発速度を維持しつつ、gameState未更新のような論理バグを自動で検出する。
ユーザーが手動テストして修正依頼する時間を削減する。

## やること

### 1. MockBukkit セットアップ

- MockBukkit の 1.21.x 対応状況を調査（非対応なら代替手段を検討）
- pom.xml にテスト依存を追加（MockBukkit, JUnit 5）
- `src/test/java/com/swarmvshunter/` にテストクラスを作成

### 2. テスト項目（requirements.md ベース）

マイルストーンごとに結合テスト的な項目を定義する。

**現時点（マイルストーン1: コマンド + フィールド生成）:**
- `/svh start` でフィールドが生成される
- `/svh start` 2回目で「ゲームが既に進行中です」と返る
- フィールドのサイズが70x70である
- 外壁（石レンガ）が配置されている
- プレイヤーがフィールド中央にテレポートされる
- プレイヤー以外からのコマンド実行が拒否される

**今後追加していくもの:**
- mob選択GUIが開く
- ゲームタイマーが動作する
- 勝敗判定が正しく動く
- Swarm変身メカニクス
- 卵システム
- etc.

### 3. 開発フローへの組み込み

CLAUDE.md に以下の方針を追加する：

```
## テスト方針

- タスク実装後は対応するテストを書く（または既存テストを更新）
- `mvn test` で全テスト通過を確認してからコミットする
- 実装完了後、Sonnetサブエージェントを起動してレビューさせる:
  1. `mvn test` 実行 → 全テスト通過確認
  2. requirements.md と実装コードの差分チェック
  3. 問題があれば報告
```

### 4. サブエージェントの動かし方

実装完了時に以下のようにSonnetエージェントを呼ぶ：

```
Agent(subagent_type="general-purpose", model="sonnet", prompt="""
以下を実行してレビューしてください:
1. mvn test を実行して全テストが通ることを確認
2. requirements.md と src/ のコードを読み比べて、未実装や論理的な漏れがないかチェック
3. 問題があれば具体的に報告。なければ「問題なし」と報告
""")
```
